# Flutter App Build Guide
## Agentic Learning App — MVP v1.0

This guide covers everything needed to build the Flutter mobile app from scratch, starting with environment setup on your **host machine** (not the devcontainer).

---

## Part 0 — Host Machine Environment Setup

Flutter cannot run inside the devcontainer — it needs the Android SDK, iOS tooling (macOS only), and hardware-accelerated emulators. Do all Flutter work on your host machine.

### 0.1 Install Flutter

```bash
# macOS (recommended)
brew install --cask flutter

# Or download directly from flutter.dev and add to PATH
# Verify:
flutter doctor
```

Target Flutter 3.22+ (stable channel). Fix everything `flutter doctor` flags before proceeding.

### 0.2 Android Setup

```bash
# Install Android Studio from developer.android.com/studio
# Inside Android Studio: SDK Manager → install:
#   - Android SDK Platform 34 (API 34)
#   - Android SDK Build-Tools 34.0.0
#   - Android Emulator
#   - Android SDK Platform-Tools

# Accept licenses
flutter doctor --android-licenses

# Create an AVD (emulator)
# Android Studio → Device Manager → Create Virtual Device
# Recommended: Pixel 7, API 34
```

### 0.3 iOS Setup (macOS only)

```bash
# Install Xcode from the App Store (latest stable)
sudo xcode-select --switch /Applications/Xcode.app/Contents/Developer
sudo xcodebuild -runFirstLaunch

# Install CocoaPods
sudo gem install cocoapods

# Open Simulator
open -a Simulator
```

### 0.4 Firebase CLI (for Analytics/Crashlytics setup)

```bash
# Install Firebase CLI
npm install -g firebase-tools

# Log in
firebase login

# Install FlutterFire CLI
dart pub global activate flutterfire_cli
```

### 0.5 Copy Firebase Config Files

The Terraform provisioning in the devcontainer already wrote these files.
Copy them from the repo to your host machine's working copy — they are git-tracked:

- `app/android/app/google-services.json` — already committed ✅
- `app/ios/Runner/GoogleService-Info.plist` — already committed ✅

No action needed if you are working from the same git clone.

### 0.6 Get the Cloud Run Backend URL

The backend is not yet deployed. Before you can run full end-to-end sessions, you need to deploy it. Steps are in [development_roadmap.md](development_roadmap.md) Phase 3.4. For local development, you can run the backend in the devcontainer and forward the port:

```bash
# In devcontainer terminal:
cd backend && uvicorn main:app --host 0.0.0.0 --port 8080

# On host machine: the devcontainer port 8080 is forwarded automatically by VS Code
# Backend URL for local dev: http://localhost:8080
```

## Part 1 — Create the Flutter Project

Run this on your host machine inside the repo root:

```bash
cd /path/to/agentic-learning-app

# Create the Flutter project in the existing app/ directory
# The directory already has google-services.json and GoogleService-Info.plist
flutter create \
  --org com.agenticlearning \
  --project-name agentic_learning_app \
  --platforms android,ios \
  app

# Verify it builds
cd app
flutter pub get
flutter run
```

> The `app/android/app/google-services.json` and `app/ios/Runner/GoogleService-Info.plist` files already exist — `flutter create` will not overwrite them.

## Part 5 — Backend API Contract

The backend exposes one endpoint for now. More will be added as agents are wired up.

### Endpoint: `POST /session/start`

**Request:**
```json
{ "uid": "<firebase_anonymous_uid>" }
```

**Response:**
```json
{ "status": "ok", "session_id": "<uuid>" }
```

The pipeline runs fully server-side. The `session_id` is for future follow-up calls (quiz answers, help turns). The full session response (lesson text, quiz questions, summary) will be streamed or returned in subsequent agent-specific endpoints — those endpoints are **not yet implemented** in the backend. The current `POST /session/start` runs the full pipeline and returns only the session ID.

> **What this means for Phase 1 of app development:** Build the app skeleton, auth flow, home screen, and character widget first. The interactive session screen (lesson text + quiz) requires the backend to expose per-turn endpoints. Coordinate with backend work in Phase 4 of the roadmap.

### Health Check: `GET /health`
```json
{ "status": "ok" }
```

### Agent Output Schemas (for future per-turn endpoints)

These are the JSON shapes each agent produces. Build your Dart model classes against these now so they're ready when the endpoints are wired.

**ContextAgent output (stored in session state as `context_output`):**
```json
{
  "next_concept_id": "L04",
  "difficulty_tier": "beginner",
  "module_character_id": "filo",
  "session_goal": "Master file navigation with cd and ls"
}
```

**LessonAgent — teaching phase:**
```json
{
  "lesson_text": "...",
  "character_emotion_state": "teaching",
  "key_concepts": ["concept1", "concept2"]
}
```

**LessonAgent — quiz question:**
```json
{
  "question_text": "Which command lists files?",
  "format": "mc",
  "options": ["ls", "cd", "pwd", "rm"],
  "character_emotion_state": "curious"
}
```

**LessonAgent — answer evaluation:**
```json
{
  "correct": true,
  "explanation": "ls lists directory contents.",
  "concept_score_delta": 0.1,
  "character_emotion_state": "celebrating",
  "trigger_help": false
}
```

**HelpAgent — resolved:**
```json
{ "resolved": true, "character_emotion_state": "celebrating" }
```

**HelpAgent — unresolved (turn 3):**
```json
{
  "resolved": false,
  "gemini_handoff_prompt": "I'm learning Linux and struggling with file permissions..."
}
```

**SummaryAgent output:**
```json
{
  "lesson_id": "L04",
  "tier_used": "beginner",
  "quiz_scores": { "L04": 0.7 },
  "time_on_task_seconds": 420,
  "help_triggered": false,
  "gemini_handoff_used": false,
  "summary_text": "...",
  "created_at": "2026-04-03T10:00:00Z"
}
```

**Valid `character_emotion_state` values:** `welcome`, `teaching`, `curious`, `celebrating`, `encouraging`, `helping`

**Module character IDs:**
| Module | character_id |
|---|---|
| 1 — What is Linux? | `tux_jr` |
| 2 — The Terminal | `cursor` |
| 3 — Files & Directories | `filo` |
| 4 — Working with Files | `snippy` |
| 5 — Users & Permissions | `keyra` |
| 6 — Processes | `spinner` |
| 7 — Networking Basics | `wavo` |

## Part 6 — Implementation Order

Build in this order. Each phase is independently testable.

### Phase A — Skeleton + Auth (no backend needed)

1. **`main.dart`** — Initialize Firebase, wrap app in `ProviderScope`, set up `MaterialApp`
2. **`auth_service.dart`** — Firebase anonymous sign-in on first launch; expose `currentUser`
3. **`auth_provider.dart`** — Riverpod `StreamProvider` watching `FirebaseAuth.instance.authStateChanges()`
4. **`splash_screen.dart`** — Show logo; wait for auth to resolve; route to Onboarding (first launch) or Home
5. **`onboarding_screen.dart`** — 3 difficulty questions; write `difficulty_tier` + `onboarding_complete: true` to Firestore; route to Home
6. **`home_screen.dart`** — Static layout: "Next Lesson" card placeholder, streak counter placeholder

**Test gate:** App launches, anonymous user is created, onboarding flow completes, home screen shows.

### Phase B — Character Widget (no backend needed)

7. **`character_widget.dart`** — Takes `characterId` + `emotionState` strings; loads `assets/characters/{id}_{emotion}.png`; uses `AnimatedCrossFade` (300ms) between emotion changes
8. **Add placeholder PNG assets** — For each of the 9 characters × 6 emotions, place a placeholder PNG at `assets/characters/{character_id}_{emotion}.png`. You can use a single colored square per character until real assets are generated (Phase 2 of roadmap)
9. Wire `CharacterWidget` into `home_screen.dart` as a preview

**Widget API:**
```dart
CharacterWidget(
  characterId: 'cursor',       // e.g. from context_output.module_character_id
  emotionState: 'teaching',   // e.g. from any agent response
  size: 80,                    // 80dp corner mode, 120dp help mode
)
```

**Test gate:** Character widget renders all 6 emotion states with cross-fade transitions. Run `flutter test test/widgets/character_widget_test.dart`.

### Phase C — Backend Client + Session Start (requires backend running)

10. **`api_config.dart`** — `const String backendUrl` (read from env or `--dart-define` for dev vs prod)
11. **`backend_client.dart`** — `Future<String> startSession(String uid)` → POST `/session/start` → returns `session_id`
12. **`session_provider.dart`** — Riverpod `StateNotifier` holding `SessionState` (sessionId, status)
13. Wire "Start Session" button on home screen → calls `backend_client.startSession()` → stores session_id

**Running with dev backend:**
```bash
# Pass the backend URL at run time
flutter run --dart-define=BACKEND_URL=http://localhost:8080
```

**Test gate:** Tap "Start Session" on home screen → session_id appears in console log. Backend logs show session created.

### Phase D — Session Screen Skeleton (requires per-turn backend endpoints)

> **Dependency:** The backend `POST /session/start` currently runs the full pipeline and returns only the session ID. Per-turn endpoints (`/session/{id}/lesson`, `/session/{id}/quiz/answer`, `/session/{id}/help`) need to be added to the backend before this phase. Coordinate with backend Phase 4.4 work.

14. **`session_screen.dart`** — Scaffold with: `CharacterWidget` pinned top-right, lesson text card, progress bar
15. **`help_bottom_sheet.dart`** — 3-turn conversation UI; slides up when `trigger_help: true`; "Got it" dismisses
16. **`session_complete_screen.dart`** — Summary text, character celebrating, next session teaser

### Phase E — Quiz Formats

17. **`multiple_choice_card.dart`** — 4-option tap-to-select list
18. **`true_false_card.dart`** — Two large toggle buttons
19. **`fill_blank_card.dart`** — Tap-to-select from 3–4 options (not free text — all formats are tap-to-select per spec)
20. **`command_card.dart`** — Tap-to-select command completion

All 4 quiz widgets take `onAnswerSelected(String answer)` callback. They do not submit — the parent `session_screen.dart` handles submission.

### Phase F — Google Sign-In Upgrade

21. **`auth_service.dart`** (extend) — `Future<void> linkWithGoogle()` — links anonymous UID to Google account; preserves all data; UID unchanged
22. Wire into a non-blocking bottom sheet on home screen; show once after session 3 completes (check Firestore `sessions` count)

### Phase G — Gemini Referral Card

23. **Inline widget in `session_screen.dart`** — Dismissible overlay card shown when `resolved: false` received from HelpAgent
24. Tap action: `url_launcher` to open Gemini app or `gemini.google.com` with `gemini_handoff_prompt` pre-filled
25. Log `gemini_handoff_tapped` Firebase Analytics event; do NOT log the prompt content

### Phase H — Analytics + Crashlytics

26. Wire `FirebaseAnalytics` events:
    - `session_start` — when session screen opens
    - `lesson_complete` — when lesson text fully displayed
    - `quiz_answer` — on each answer submitted (log `correct: bool` only, not question text)
    - `help_triggered` — when HelpAgent activates
    - `gemini_handoff_tapped` — when user taps Gemini referral card
    - `session_complete` — when session complete screen loads
27. Wire `FirebaseCrashlytics.instance.recordFlutterFatalError` in `main.dart`

## Part 7 — Character Assets

Character PNGs are not yet generated (Phase 2 of roadmap). During development:

1. Create placeholder assets using any image tool — a simple colored PNG per character is fine for testing the widget
2. Name files exactly: `{character_id}_{emotion}.png`
   - Example: `cursor_teaching.png`, `cursor_celebrating.png`
3. Place all files in `app/assets/characters/`
4. The `pubspec.yaml` already declares `assets/characters/` as the asset directory

When real assets are generated (Phase 2), replace the placeholders — no code changes needed.

**Full asset list (54 files = 9 characters × 6 emotions):**
```
{tux_jr|cursor|filo|snippy|keyra|spinner|wavo|boxby|scrippy}_{welcome|teaching|curious|celebrating|encouraging|helping}.png
```

**Placeholder creation (bash — run on host):**
```bash
cd app/assets/characters
for char in tux_jr cursor filo snippy keyra spinner wavo boxby scrippy; do
  for emotion in welcome teaching curious celebrating encouraging helping; do
    # Create a 64x64 white PNG as placeholder (requires ImageMagick)
    convert -size 64x64 xc:lightblue "${char}_${emotion}.png"
  done
done
```
Or simply copy a single test image 54 times with the correct names.

## Part 8 — Configuration

### Dev vs Production Backend URL

```dart
// lib/config/api_config.dart
class ApiConfig {
  static const String backendUrl = String.fromEnvironment(
    'BACKEND_URL',
    defaultValue: 'http://localhost:8080',
  );
}
```

Run with:
```bash
# Local dev (backend in devcontainer, port-forwarded)
flutter run --dart-define=BACKEND_URL=http://localhost:8080

# Production (after Cloud Run deploy)
flutter run --dart-define=BACKEND_URL=https://<cloud-run-url>
```

For release builds, bake in the production URL:
```bash
flutter build apk --dart-define=BACKEND_URL=https://<cloud-run-url>
```

### Firebase Auth Token Injection

Every request to the backend should carry the Firebase ID token for authentication once the backend adds auth middleware (Phase 4+). Wire it now so it's ready:

```dart
// In backend_client.dart
final user = FirebaseAuth.instance.currentUser;
final token = await user?.getIdToken();
// Add to headers: 'Authorization': 'Bearer $token'
```
